## 🐛 缺陷清單

### 🔴 BUG-1（P1）— `/status/{taskId}` 二次查詢回傳 500

| 項目 | 說明 |
|------|------|
| **端點** | `GET /api/face-swap/status/{taskId}` |
| **問題** | 任務首次轉為 completed 後，**再次查詢回傳 HTTP 500** |
| **根因** | `FaceSwapController::checkStatus()` 的 `Cache::lock callback` 在 avatar 已 completed 時未 return，外層 `$avatar = null` → NullPointerException |
| **影響** | 前端輪詢到 completed 後若再查詢（如重新整理頁面），將看到 500 錯誤 |
| **建議修復** | `FaceSwapController.php` callback 末尾加 `return $avatar;` |
| **重現** | `GET /api/face-swap/status/1875`（已 completed 的 task） |

### 🟡 BUG-2（P2）— `/status` 回應結構與 submit 不一致

| 項目 | 說明 |
|------|------|
| **端點** | `GET /api/face-swap/status/{taskId}` |
| **問題** | `/status` 回應有 middleware 包裝層，`images` 在 `result.images`；但 POST submit 的 `task_id` 在 `result.task_id` 直接可取 |
| **影響** | 前端若統一用 `response.images` 取圖會得到 undefined，需從 `response.result.images` 取 |
| **建議** | 統一 API 回應結構，或在 API 文件明確標注層級差異 |

## 關鍵觀察

1. **Submit API 極快**（0.39–0.55s），生成時間瓶頸完全在 AI 處理端
2. **並發不劣化，反而更快**：2 並發 avg 36.5s、3 並發 avg 31.8s，優於單次 avg 39.7s
   → AI Server（60.248.142.147:7588）具備平行處理能力，GPU 在多任務時效率更高
3. **生成時間穩定**：全 14 筆範圍 31.8s–41.9s，標準差約 3.5s，無異常離群值
4. **缺乏優化前基準值**：無法驗證「縮短幅度」，建議向 RD 取得舊版本數據比對
